## Operator action items (non-PR)

1. **Toggle `enforce_admins: True`** on the `main` branch protection rule.
   Read-back via `gh api repos/KLYTICS/cerniq/branches/main/protection`
   confirmed the protection IS configured (`required_pull_request_reviews:
   present` with 1 approval required, `required_linear_history: True`,
   `allow_force_pushes: False`), but **`enforce_admins: False`** means
   admins bypass *all* of it. That's how `75cd425`, `3d4059d`, `3bdd515`
   all landed directly on main without going through PRs.
   - **GitHub UI:** Settings → Branches → main → Edit → check
     **"Do not allow bypassing the above settings"**.
   - **gh CLI:** `gh api -X POST repos/KLYTICS/cerniq/branches/main/protection/enforce_admins`
     (REST: **POST** to enable, **DELETE** to disable. The
     `--method PATCH` form previously documented here was wrong —
     this endpoint is a single-purpose toggle, not a partial-update.
     Verified against GitHub's branch-protection REST docs;
     correction caught by swarm-2 code-reviewer 2026-05-27.)
   - One toggle. Until it flips, the PR-based mirror's safety property
     is convention-only.
2. **Resolve the rad/main ↔ origin/main divergence** (6 commits ahead on
   GitHub never propagated to Radicle — see SESSION_HANDOFF
   2026-05-27 entry). The mirror cron now correctly refuses to open a
   destructive PR for this state, surfacing it as a `::warning::` per
   the divergence-detection logic in [PR #63 / commit `3bdd515`].
3. **Decide on PR #61's CVE-patches re-land.** The original attempt was
   reverted post-merge (`bdccdd9`); the underlying CVE remediation still
   matters.
4. **Decide on PR #12's philosophical approach** (fail-fast on missing
   secrets vs gracefully-skip per the landed `3d4059d`). See inline
   comment on PR #12 for full trade-off matrix.

5. **Plan Tier-2 architectural follow-ups from swarm-2** (the
   substantive items, deferred from PR #58 + PR #64 to keep those PRs
   scoped):
   - **Audit-outbox is vaporware** (highest leverage). `ARCHITECTURE.md`
     §10.4 advertises an audit outbox for PG-down survival; the actual
     `OutboxKind` enum has only `BATE_SIGNAL | WEBHOOK_DELIVERY`.
     `verify.algorithm.ts:237-240` silently catches audit-append errors
     on the deny path — closing this hole + adding `AUDIT_APPEND` to
     `OutboxKind` is the single biggest invariant-vs-implementation
     gap in the codebase. Closes invariants #3 AND #4 simultaneously.
   - **Verify-algorithm portability CI guard**. Invariants #2 and #8
     ("/v1/verify portable", "SDKs portable") are policed only by
     code review today. Add a `rg -l '@nestjs|@prisma|bullmq|node:fs|
     node:crypto' apps/api/src/modules/verify/algorithm/` step that
     fails on any match. Same for `packages/sdk-ts/`.
   - **Transactional revocation**. `identity.service.revoke()` and
     `policy.service.revoke()` do state flip + audit append +
     webhook enqueue as 3 separate calls. A process crash between any
     two leaves a revoked agent with no chain entry / no subscriber
     notification. Wrap in `prisma.$transaction` + use outbox for
     webhook fan-out.
   - **Branded ID types**. `PrincipalId`, `AgentId`, `PolicyId`,
     `ApiKeyId`, `WebhookSubscriptionId`, `AuditEventId`, `SigningKeyId`,
     `RelyingPartyId` are all raw `string` today. Brand via Zod
     `.brand<...>()` + derived TS types; service signatures become
     self-checking against the apiKeyId/reason swap risk the first
     swarm caught on PR #58.
   - **SDK shadow types**. `packages/sdk-ts/src/types.ts:1-31`
     hand-rolls `AgentRuntime`, `SpendLimit.currency`, `ScopeCategory`
     in a way that drifts from `@cerniq/types` (different casing,
     missing currencies). Re-export from `@cerniq/types` and delete
     the shadow copies. Add a parity spec.
   - **Policy expiry audit-append**. `policy.expiry.worker` sweeps
     flip status to REVOKED with NO `audit.append()` — implicit
     revocations lack signed SOC2 evidence. Tiny fix after PR #58
     merges (PR #58 wires AuditModule into PolicyModule).

   Tier-2 items are NOT blocking PR #58 / #62 / #63 / #64 merges.
   They're substantive enough each warrants its own PR.
