The Problem

AI agents are acting without accountability

Autonomous agents are being deployed to call APIs, manage data, and execute decisions on behalf of users — with no standard way to verify who authorized the action, whether the policy permitted it, or whether the audit trail is trustworthy.

The agentic era needs the same infrastructure the web got in the 1990s: a universal trust anchor that every relying party can verify independently.

How It Works

Three steps to verified agent trust

From key registration to real-time verification, OKORO sits in the critical path without being in the critical path.

1
Register
The agent generates an Ed25519 keypair client-side. The public key and policy are registered with OKORO. The private key never leaves the agent.
POST /v1/agents
2
Sign & Act
Before each action, the agent signs the request payload with its private key. OKORO issues a short-lived JWT attesting the agent's identity and policy scope.
POST /v1/verify
3
Attest
The relying party verifies the JWT. OKORO appends a signed audit event to the chain. BATE updates the agent's trust score asynchronously.
append AuditEvent
Architecture Principles

Non-negotiable invariants

These are the architectural guarantees OKORO makes to every relying party. They are published, versioned, and cannot be changed without a major version bump.

01

Private keys never enter OKORO

Agent private keys are generated client-side. Our database stores public keys only. The SDK is the only surface that touches a private key, and only locally.

02

The verify hot path is portable

All signature, policy, and spend evaluation logic has zero framework imports. This is what lets us migrate /v1/verify to Cloudflare Workers without a rewrite.

03

The audit log is append-only and signed

Every write goes through audit.service.append(). No UPDATE or DELETE on AuditEvent ever. Each event signs over the previous event's signature.

04

No silent failures, no fabricated data

If a downstream call fails, we surface it. We never return a synthetic trust score, a stub policy, or an empty array that pretends to be "no results."

05

Multi-tenant isolation on every query

Every service method takes principalId as its first argument. Cross-principal data leaks are architecturally impossible at the query level.

06

Denial precedence is fixed and published

AGENT_NOT_FOUND → REVOKED → INVALID_SIG → POLICY_REVOKED → EXPIRED → SCOPE → SPEND → TRUST_SCORE → ANOMALY. Relying parties code against this order.
