## The pattern

MCP tool calls are the natural early-adopter wedge for CERNIQ:

1. The LLM picks a tool to call.
2. The MCP server receives the call.
3. **Before invoking the downstream API**, the MCP server calls
   `cerniq.verify()` with a token signed by the agent's keypair.
4. If the verify fails, the MCP server returns a structured tool error to
   the LLM — which will see it as a regular tool failure and retry or escalate.

## Wiring via `@cerniq/mcp-server`

```ts
import { createServer } from '@cerniq/mcp-server';

const server = createServer({
  cerniq: {
    apiKey: process.env.CERNIQ_API_KEY!,
    verifyKey: process.env.CERNIQ_VERIFY_KEY!,
  },
  tools: [
    {
      name: 'send-email',
      description: 'Send a transactional email',
      verifyAction: 'send_email',
      handler: async (args, { verifyToken }) => {
        // verifyToken was already verified by the CERNIQ MCP server
        // wrapper — your handler runs only on `allowed = true`.
        await emailService.send(args);
        return { sent: true };
      },
    },
  ],
});

server.listen({ stdio: true });
```

The wrapper handles token verification, replay defense, and denial-reason
mapping to MCP-shaped error responses automatically. The denial reasons
follow the [precedence chain](/docs/concepts/denial-precedence) — the LLM
can be prompted to interpret them and choose a different tool or escalate.

## Claude Desktop config

Add this entry to your `claude_desktop_config.json`:

```json
{
  "mcpServers": {
    "cerniq-protected-tools": {
      "command": "npx",
      "args": ["@cerniq/mcp-server"],
      "env": {
        "CERNIQ_API_KEY": "ak_live_...",
        "CERNIQ_VERIFY_KEY": "vk_live_..."
      }
    }
  }
}
```

## MCP-bridge alternative

If you already have an MCP server you'd rather not rewrite, wrap it with
[`@cerniq/mcp-bridge`](https://github.com/klytics/cerniq/tree/main/packages/mcp-bridge)
instead — same verify wrapping, zero changes to your existing tool handlers.
