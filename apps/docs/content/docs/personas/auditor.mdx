OKORO is designed to be auditable end-to-end. This page indexes the controls
and evidence artifacts external auditors and compliance counsel will request.

## SOC2 Type II — what to ask OKORO for

| Trust Service Criterion | Evidence OKORO produces                                                                                  |
| ----------------------- | -------------------------------------------------------------------------------------------------------- |
| CC6.1 (logical access)  | Bcrypt-hashed API keys with FULL / VERIFY-ONLY scope; principal-scoped tenant isolation enforced in service layer; tested in `tests/cross-package/principal-isolation.spec.ts`. |
| CC6.6 (encryption)      | TLS-only ingress; KMS-backed audit signing key; envelope-encrypted Ed25519 audit key on AWS until KMS GAs EdDSA. |
| CC7.2 (anomaly detection) | BATE engine R-1…R-5 rule-based detector (velocity, geographic, spend pattern, failed-verify spike, delegation depth). |
| CC7.4 (external comm)   | Status page (`status.okoroapp.com`) + incident process documented in `docs/INCIDENT_RESPONSE.md`.        |
| CC8.1 (change mgmt)     | Append-only Prisma migrations; CI gate (`spec-sync.yml`) on contract drift; PR-only deploys.             |

## GDPR — controller obligations

| Right                | Endpoint                                            | Notes                                                                                                              |
| -------------------- | --------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| Art. 15 — access     | `GET /v1/audit/export.ndjson?agentId=…`             | Streams the relevant audit slice; signed and replayable offline                                                    |
| Art. 17 — erasure    | `POST /v1/compliance/audit/redact-event`            | Nulls PII fields in the row body; signature continues to cover `decisionReasonHash`, so chain integrity is preserved |
| Art. 20 — portability | NDJSON export                                       | Same endpoint as Art. 15; consumed by `@okoro/verifier-rp` for offline replay                                       |

The redact endpoint is `ApiKeyGuard`-protected and requires **FULL** scope.
A meta-row noting the redaction (who, when, which row) is itself appended to
the chain — the act of redacting is auditable.

## EU residency

OKORO supports a single-region EU posture: API, database, and KMS all
pinned to `eu-west-1` (or operator-specified region). See
[`docs/EU_RESIDENCY.md`](https://github.com/klytics/okoro/blob/main/docs/EU_RESIDENCY.md).

## Retention horizon

Default audit retention: 7 years (SOC2 Type II floor; matches financial-
services audit norms). Operator-configurable per environment via
`OKORO_AUDIT_RETENTION_DAYS`. See [OD-004 in `OPERATOR_DECISIONS.md`](https://github.com/klytics/okoro/blob/main/OPERATOR_DECISIONS.md).

## Data Protection Addendum

A DPA template lives at [`docs/COMPLIANCE.md`](https://github.com/klytics/okoro/blob/main/docs/COMPLIANCE.md).
It is structured around the GDPR Art. 28 processor clauses and references
the OKORO sub-processors (database host, KMS provider, observability vendor).
