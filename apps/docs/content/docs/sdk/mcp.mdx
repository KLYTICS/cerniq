CERNIQ ships two MCP packages:

| Package               | Use it when…                                                                              |
| --------------------- | ----------------------------------------------------------------------------------------- |
| `@cerniq/mcp-server`   | You're building a new MCP server. Get a verified-by-default tool wrapper API.            |
| `@cerniq/mcp-bridge`   | You already have an MCP server and want to wrap it without rewriting any tool handlers.  |

## `@cerniq/mcp-server`

### Install

```bash
npm install @cerniq/mcp-server
```

### Define verified tools

```ts
import { createServer } from '@cerniq/mcp-server';

const server = createServer({
  cerniq: {
    apiKey: process.env.CERNIQ_API_KEY!,
    verifyKey: process.env.CERNIQ_VERIFY_KEY!,
  },
  tools: [
    {
      name: 'send-email',
      description: 'Send a transactional email',
      verifyAction: 'send_email',
      schema: { /* JSON Schema */ },
      handler: async (args, { agentId, principalId, verifyResult }) => {
        // Runs only on `verifyResult.allowed === true`. The wrapper has
        // already walked the denial precedence and returned a structured
        // tool error if the verify failed.
        await emailService.send(args);
        return { sent: true };
      },
    },
  ],
});

server.listen({ stdio: true });
```

### Claude Desktop config

```json
{
  "mcpServers": {
    "cerniq-protected-tools": {
      "command": "npx",
      "args": ["@cerniq/mcp-server"],
      "env": {
        "CERNIQ_API_KEY": "ak_live_...",
        "CERNIQ_VERIFY_KEY": "vk_live_..."
      }
    }
  }
}
```

## `@cerniq/mcp-bridge`

Wrap an existing MCP server with verification middleware. No tool-handler
changes required:

```ts
import { createMcpBridge } from '@cerniq/mcp-bridge';
import { myExistingMcpServer } from './my-server';

export default createMcpBridge({
  upstream: myExistingMcpServer,
  cerniq: {
    apiKey: process.env.CERNIQ_API_KEY!,
    verifyKey: process.env.CERNIQ_VERIFY_KEY!,
  },
  toolActionMap: {
    'send-email': 'send_email',
    'charge-card': 'charge',
  },
});
```

The bridge intercepts every tool call, extracts the authority token from
the request envelope (`_cerniq_token`), calls `cerniq.verify()`, and only
forwards to the upstream tool on `allowed: true`. Denials translate to
structured MCP tool errors that LLMs can interpret.

## Audit attribution

When either package calls `/v1/verify`, it stamps the relying-party id
into the resulting audit row. The dashboard's MCP-server list view shows
recent invocation counts and denial rates per server — see
[ADR-0008](https://github.com/klytics/cerniq/blob/main/docs/decisions/0008-mcp-as-control-plane.md).
